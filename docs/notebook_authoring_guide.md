# 실습 노트북 작성 가이드

이 문서는 데이터 실습용 원본 노트북과 스켈레톤 노트북을 만들 때 재사용할 수 있는 작성 원칙을 정리한 것입니다.

특정 주차나 특정 데이터셋에 묶이지 않고, 이후 다른 실습을 만들 때도 같은 기준으로 적용합니다.

## 1. 원본 노트북 작성 원칙

### 먼저 풀 질문을 제시한다

실습 초반에 이번 시간에 답할 질문을 먼저 둡니다.

좋은 형태:

```text
이번 시간에 풀 질문
1. 어떤 기준으로 신뢰할 만한 데이터를 고를 수 있을까?
2. 어떤 그룹이 더 높은 성과를 보일까?
3. 시간에 따라 어떤 패턴이 달라질까?
4. 결과 테이블은 어떤 형태가 읽기 좋고, 어떤 형태가 후속 처리에 좋을까?
```

질문을 먼저 제시하면 학생이 뒤에서 나오는 정제, 조인, 파생 컬럼, 집계, 피벗을 “왜 하는지” 이해하기 쉽습니다.

피해야 할 흐름:

```text
일단 컬럼을 만들고, 나중에 이 컬럼을 어디에 쓸지 설명한다.
```

실습 코드는 질문에 답하기 위한 과정이어야 합니다. 목적 설명이 코드 중간에 처음 등장하면 흐름이 약해집니다.

### 테이블 단계의 역할을 구분한다

데이터 실습은 보통 다음 흐름으로 구성합니다.

```text
raw
→ intermediate
→ output / business table
```

각 단계의 역할을 명확히 합니다.

- raw: 아직 손대지 않은 원본 데이터
- intermediate: 정제, 표준화, 조인, 파생 컬럼 생성이 끝난 재사용 가능한 중간 테이블
- output / business table: 특정 질문에 답하기 위해 집계, 피벗, 재구조화된 결과 테이블

학생이 지금 만드는 테이블이 “저장/재사용용”인지 “보고/분석용”인지 구분할 수 있어야 합니다.

### 정제 과정은 자연스러운 품질 점검 흐름으로 설명한다

결측 처리, 중복 제거, 자료형 변환, 키 표준화는 특별한 예외 상황처럼 설명하지 않습니다.

좋은 흐름:

```text
clean table 만들기
- 필요한 컬럼만 선택
- key 컬럼 이름 표준화
- key 컬럼 자료형 변환
- 필요한 파생 컬럼 생성
- 결측치와 중복 확인
- 분석 단위에 맞게 1행의 의미 정리
```

피해야 할 흐름:

```text
중요한 점이 하나 더 있습니다.
이걸 안 하면 큰 문제가 생깁니다.
```

이런 표현은 학습 흐름을 끊습니다. 정제는 데이터 품질을 맞추는 기본 절차로 설명합니다.

### 조인은 현재 데이터의 관계에 맞게 설명한다

조인 종류를 일반론으로 길게 설명하기보다, 현재 데이터에서 어떤 관계를 만들고 싶은지 먼저 말합니다.

확인할 질문:

- 기준 테이블은 무엇인가?
- 조인 키는 무엇인가?
- 조인 후에도 분석 단위가 유지되는가?
- 결측은 조인 문제인가, 이후 필터링 문제인가?

중요한 구분:

```text
join 조건: key가 양쪽 테이블에 있는가
분석 조건: 분석에 필요한 값이 비어 있지 않은가
```

분석 조건을 조인 방식으로 해결하려고 하면 설명이 꼬일 수 있습니다. 값의 결측은 필요할 때 필터링 단계에서 다룹니다.

피해야 할 예:

```text
참고로 inner join을 쓰면 얼마나 줄어드는지 비교만 해봅니다.
```

현재 데이터에서 실제 차이가 없거나 세션 목표와 맞지 않으면 비교 셀은 제거합니다.

### 부연 개념은 참고 섹션으로 분리한다

본 흐름에서 벗어나지만 알아두면 좋은 개념은 번호 섹션으로 만들지 않고 `참고`로 둡니다.

예시:

```text
### 참고: explode는 무엇인가?
```

참고 섹션은 짧아야 합니다. 그 개념을 직접 구현하지 않는다면 정의와 왜 이번 실습에서는 단순화하는지만 설명합니다.

### 결과가 읽기 어렵다면 표시 방식을 조정한다

큰 숫자, 날짜, 비율처럼 초보자가 읽기 어려운 출력은 표시 방식을 조정합니다.

예시:

```python
pd.options.display.float_format = "{:,.2f}".format
```

목표는 정확도 자체를 바꾸는 것이 아니라, 실습 중 결과를 해석하기 쉽게 만드는 것입니다.

### 예제의 단위를 조정한다

피벗이나 집계 결과가 너무 sparse하면 학습 효과가 떨어집니다.

예시:

```text
일별 x 카테고리
→ 월별 x 카테고리
```

```text
연도 x 키워드
→ 연대 x 키워드
```

결과 테이블이 비어 있는 값으로 가득 차면 함수 사용법보다 데이터 구조의 노이즈가 더 커집니다. 필요한 경우 시간 단위, 카테고리 수, 상위 N개 기준을 조정합니다.

### 시각화는 세션 목표에 맞게 역할을 제한한다

시각화 자체가 목표가 아니라면 코드는 제공하고, 결과 차이를 보여주는 용도로 둡니다.

좋은 설명:

```text
아래 시각화 코드는 직접 작성하는 대상이라기보다,
long format이 왜 시각화에 편한지 보여주기 위한 예시입니다.
```

시각화 코드까지 학생이 직접 작성하게 할지, 결과만 확인하게 할지는 세션 목표에 따라 정합니다.

### 한글 폰트와 환경 차이를 처리한다

한글 제목이나 라벨이 깨질 수 있는 실습에서는 노트북 초반에 폰트를 설정합니다.

예시:

```python
font_candidates = ["AppleGothic", "Malgun Gothic", "NanumGothic"]
available_fonts = {font.name for font in font_manager.fontManager.ttflist}

for font_name in font_candidates:
    if font_name in available_fonts:
        rcParams["font.family"] = font_name
        break

rcParams["axes.unicode_minus"] = False
```

환경별 차이가 있는 설정은 학생이 직접 채울 부분으로 두지 않는 것이 좋습니다.

## 2. 스켈레톤 노트북 작성 원칙

### 세션 핵심 포인트만 비운다

스켈레톤은 코드 빈칸 채우기 문제집이 아닙니다.

세션 목표와 직접 연결되는 부분만 최소한으로 `TODO` 처리합니다.

예를 들어 Transform 심화 실습이라면 다음 정도만 비웁니다.

- join
- filtering
- `groupby + agg`
- `pivot_table`
- `melt`

채워두는 것이 좋은 부분:

- 경로 설정
- 데이터 로딩
- 폰트 설정
- 출력 포맷 설정
- 복잡한 문자열 파싱 함수
- 반복적인 정제 코드
- 단순 파생 컬럼 생성
- 시각화 코드
- 저장 코드

### 구현 디테일은 가능하면 채워둔다

세션 목표와 직접 관련 없는 구현 디테일은 비우지 않습니다.

채워두는 것이 좋은 예:

```python
def parse_nested_text(value):
    ...
```

```python
df["ratio"] = np.where(...)
```

이런 부분을 비우면 학생의 시선이 핵심 개념이 아니라 Python 구현 세부사항으로 분산됩니다.

### TODO에는 결과물의 모양을 알려준다

단순히 “집계한다”라고만 쓰지 말고, 기대 결과 컬럼과 변수명을 주석으로 알려줍니다.

좋은 예:

```python
# TODO: reliable_items를 year 기준으로 집계한다.
# 결과물 이름은 yearly_summary로 둔다.
# 결과 컬럼은 아래 이름을 사용한다.
# - item_count: 항목 수
# - avg_score: 평균 점수
# - avg_review_count: 평균 리뷰 수
# 힌트: groupby("year") + agg(...) + reset_index() + sort_values(...)
yearly_summary = None
```

이렇게 하면 학생이 결과 테이블의 구조를 먼저 떠올리고 코드를 작성할 수 있습니다.

### TODO 개수는 적게 유지한다

TODO가 많으면 실습 목표가 흐려집니다.

권장 기준:

- 짧은 실습: 3~5개
- 한 세션짜리 실습: 5~8개
- 심화 실습: 8~12개 이하

TODO가 많아진다면 핵심 개념과 구현 디테일이 섞였을 가능성이 큽니다.

### 스켈레톤은 출력 없는 상태로 저장한다

스켈레톤 노트북은 실행 결과가 없는 상태로 둡니다.

확인 기준:

```text
outputs: 0
execution_count: 0
```

원본 노트북은 실행 결과를 남겨도 되지만, 스켈레톤은 학생이 직접 실행하며 결과를 확인해야 합니다.

## 3. 최종 점검 체크리스트

원본 노트북:

- 초반에 이번 실습의 질문이 제시되어 있는가
- raw → intermediate → output 흐름이 보이는가
- 각 테이블의 역할이 명확한가
- 조인 설명이 실제 데이터 관계와 맞는가
- 결측 필터링과 조인 조건이 섞여 있지 않은가
- 불필요한 비교 셀이나 오래된 설명이 남아 있지 않은가
- 숫자, 날짜, 비율 출력이 읽기 쉬운가
- 피벗/집계 결과가 지나치게 sparse하지 않은가
- 시각화 코드는 세션 목표에 맞는 역할만 하는가
- 환경 차이로 깨질 수 있는 설정은 미리 처리되어 있는가

스켈레톤 노트북:

- 세션 핵심과 무관한 구현 디테일은 채워져 있는가
- TODO가 너무 많지 않은가
- 각 TODO에 결과 변수명과 기대 컬럼명이 적혀 있는가
- 출력과 실행 번호가 비워져 있는가
- 완성본과 같은 흐름을 유지하는가
